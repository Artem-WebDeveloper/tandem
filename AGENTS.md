# Repository Guidelines

## Структура Проекта

Это приложение на `Vite + React 19 + TypeScript`. Основной код находится в `src/`.

- `src/app` содержит каркас приложения и точки входа роутинга.
- `src/pages` содержит страницы уровня маршрутов: `Dashboard`, `Library`, `Practice`, `Results` и другие.
- `src/core` содержит общие части приложения: `components`, `feature`, `api`, `store` на Zustand, `utils`, `configs`, `types`, `theme`, `i18n`.
- `public` хранит публичные статические файлы, а переиспользуемые ресурсы приложения лежат в `src/core/assets`.
- Тесты располагаются рядом с исходниками и называются по шаблону `*.test.tsx`.

Новые UI-части лучше добавлять в рамках конкретной страницы или фичи, а не в виде новых общих верхнеуровневых папок.

## Команды Сборки, Тестов И Разработки

- `npm run dev` запускает локальный dev-сервер Vite.
- `npm run build` проверяет TypeScript-проект и собирает production-бандл.
- `npm run preview` локально открывает собранную версию приложения.
- `npm run lint` запускает ESLint по всему репозиторию.
- `npm run lint:fix` автоматически исправляет часть lint-ошибок в `ts` и `tsx`.
- `npm run format` форматирует проект через Prettier.
- `npm run ci:format` проверяет форматирование без изменений файлов.
- `npm test` запускает тесты в Vitest.

Перед созданием PR минимум запускайте `npm run lint` и `npm test`.

## Стиль Кода И Именование

Используйте отступ в 2 пробела, `LF`, `UTF-8`, точки с запятой, одинарные кавычки и завершающие запятые. Базовые правила задаются в `.editorconfig` и `.prettierrc`.

Принятые соглашения:

- `PascalCase` для React-компонентов и папок страниц, например `PracticeHeader.tsx`.
- `camelCase` для утилит, стора и вспомогательных модулей, например `loginValidation.ts`.
- `*.module.scss` для локальных стилей компонентов.

ESLint настроен для `ts` и `tsx`. Не отключайте правила без явной необходимости.

## Правила Тестирования

В проекте используется `Vitest` вместе с Testing Library; конфигурация находится в `vitest.config.ts`, общее тестовое окружение в `src/setupTests.ts`.

Пишите тесты рядом с компонентом или модулем. Основной фокус: рендеринг, пользовательские сценарии, навигация, валидация и переходы состояния. Если меняется поведение виджета или общего UI-компонента, тесты нужно добавить или обновить.

## Коммиты И Pull Request

История репозитория использует короткие сообщения в стиле Conventional Commits, например `fix: add react list key` и `refactor: code ordering store cleanup`. Предпочтительные префиксы: `feat:`, `fix:`, `refactor:`, `test:`, `docs:`.

Перед коммитом срабатывает `npx lint-staged` через Husky, поэтому индекс должен быть чистым по форматированию и линтингу. В PR желательно включать:

- краткое описание изменения и затронутой области;
- ссылку на задачу или issue, если она есть;
- скриншоты или видео для визуальных изменений;
- пометку о добавленных тестах или известных пробелах в покрытии.
