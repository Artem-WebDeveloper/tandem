# Проектирование архитектуры квизов и работа над UI

## Задачи:

* Исправить баг со стилями в форме авторизации при автозаполнении.
* Спроектировать и реализовать систему моделей для квизов разного типа.
* Создать сериализаторы и представления (Views) для работы с вопросами через API.
* Начать первичное наполнение базы данных.

## Ход работы:

#### Мелкие правки UI

Мне указали в pr, что при использовании браузерного автозаполнения (логин/пароль) стандартные стили `Material UI` перекрывались системными цветами браузера. Добавила CSS-правила, чтобы сохранить визуальную целостность формы при автоматическом вводе данных.

#### Архитектура моделей квизов

Это был самый объёмный блок задач. Чтобы не усложнять код, я решила разбить модели на составляющие части:

1. **Базовые модели:** самые простые модели с 1-2 полями (сложность, тема, теги и разделы).
2. **Общая модель квиза:** связующее звено, которое объединяет вопросы в логические группы.
3. **Специфичные модели:** модели для двух разных типов квизов. Каждая модель расширяет базовую своими уникальными полями.

Такой подход позволит в будущем легко добавлять новые форматы квизов, не переписывая всю логику бэкенда.

#### API и база данных

Для новых моделей были написаны сериализаторы в `Django Rest Framework`. Особое внимание уделила вложенности: при запросе всех доступных квизов выводится краткая информация, а при запросе конкретного квиза фронтенд должен получать не только заголовки, но и список всех относящихся к нему вопросов.

Также реализовала соответствующие `Views` и зарегистрировала их в `urls.py`. Чтобы проверить работоспособность, я частично внесла тестовые данные напрямую в базу. Пока только не добавляла требование на авторизацию для запроса. Сделаю это чуть позже.

**Итог:** Система квизов наконец отдает первые данные через API. Дальше буду расширять типы квизов и наполнение бд, а также добавлю фильтрацию для сложных запросов.
