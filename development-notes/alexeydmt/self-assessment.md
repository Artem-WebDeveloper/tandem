# Описание работы

В рамках проекта я занимался тремя направлениями: архитектурой системы локализации и типизированных ошибок, UX-слоем страницы логина/регистрации и компонентом Single Choice Quiz.

## Основные задачи и вклад

### 1. Архитектура i18n — система локализации приложения

Спроектировал и внедрил систему локализации. В рамках этой задачи затронуто множество файлов, далее команда подхватила архитектуру и интегрировала переводы в дальнейшие разрабытываемые компоненты.

Ключевые решения:

- Был выбран **namespace-подход** вместо одного большого файла — 7 пространств имён (common, dashboard, library, login, practice, not-found, results). Это позволило команде работать параллельно без конфликтов и чётко разделить зоны ответственности;
- подключил `i18next-browser-languagedetector` — автоопределение языка браузера с кешированием в `localStorage`, выбранный язык не теряется при перезагрузке;
- написал хук `useLocale()` — квизы хранят контент с двумя языками прямо в объекте данных (`question.text.ru / .en`), это не i18next, а поле из API. `useLocale()` возвращает `'ru' | 'en'` (а не `'ru-RU'`) именно для обращения к таким полям;
- настроил `fallbackLng`, `defaultNS`, `escapeValue: false`.

### 2. Архитектура типизированных ошибок (`AppError`)

Столкнулся с архитектурной проблемой: ошибки бросаются в API-слое (чистые функции без React-контекста), но отображать их нужно локализованно в компонентах. Стандартный подход `throw new Error('строка')` ломается при добавлении i18n.

Спроектировал решение:

- `AppErrorCode` — перечисление через `as const`;
- `AppError extends Error` с полями `code` и `params` — `params` нужен для интерполяции: `errors.quizNotFound` с `{ id }` превращается в `"Квиз с id 42 не найден"`;
- заменил все `throw new Error('Какая-то ошибка')` во всех файлах, существующих на момент создания `AppError`.

### 3. Система валидации форм

Разработал утилитарный модуль `loginValidation.ts`.

- `validateUsername` — длина (3–15), паттерн символов, ранний выход при пустом поле;
- `validatePassword` — минимальная длина, сравнение с логином, чёрный список распространённых паролей, запрет паролей только из цифр;
- `getPasswordStrength` — проверка по 5 параметрам → `weak / medium / strong`, результат используется в `LinearProgress` как `(score / 5) * 100`;
- функции возвращают **массив i18n-ключей**, UI сам решает на каком языке отобразить ошибку.

### 4. UX-слой Login страницы и RegisterDialog

Базовая логика отправки форм существовала до меня. Моя задача — привести страницу к общему стилю и функционалу приложения, перенос большей части валидации на frontend и другие нюансы (https://github.com/Artem-WebDeveloper/tandem/issues/45):

- интегрировал систему валидации: real-time проверка после первого касания поля (touched-логика через `useEffect` + флаги состояния);
- визуальный индикатор силы пароля с цветовой градацией (красный → жёлтый → зелёный);
- показ/скрытие пароля через `InputAdornment`;
- `Collapse`-анимации для алертов — ошибки появляются и исчезают плавно;
- ограничение ввода логина до 15 символов прямо в `handleChange` + счётчик символов в `helperText`;
- автоочистка ошибок через `setTimeout`;
- подключил весь текстовый слой к i18n (`login` namespace).

### 5. Single Choice Quiz компонент

Реализовал компонент квиза:

- спроектировал типы: `SingleChoiceTaskResponse`, `SingleChoiceQuestion`, `SingleChoiceOption` с поддержкой мультиязычности (`text: { ru, en }`);
- хранение ответов в массиве по индексу вопроса, позволяющее восстанавливать ответ при возврате назад;
- визуальный фидбэк выбранного варианта (лёгкое масштабирование через `transform: scale`, плавная анимация);
- использовал `useLocale()` для отображения текста вопросов и вариантов на нужном языке;

### 6. Архитектура типов квизов (`quiz.ts`)

Участвовал в проектировании базовой типовой системы:

- `BaseTask` — общий интерфейс для всех квизов;
- `TaskType` через `as const`;
- `QuizTask` — union всех типов, используется в `fetchQuizById` и `Practice.tsx`.

## Основные сложности

- **Перевод ошибок из API-слоя** — `i18next` недоступен вне React-компонентов, а ошибки нужно бросать в чистых функциях. `AppError` с кодом и параметрами — решение, которое масштабируется на весь проект;
- **Логика "touched"** — баланс между "не спамить ошибками сразу" и "показать все проблемы когда нужно".
- **Две системы переводов** — `i18next` для интерфейса и `text: { ru, en }` для контента из БД.
