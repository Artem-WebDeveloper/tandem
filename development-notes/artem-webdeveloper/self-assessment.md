## 2. Описание работы

### Роль и организация

Я выступал в роли тимлида без опыта командной разработки. С первого дня взял на себя организацию workflow: предложил структуру веток, описал правила ведения разработки в команде, чтобы участникам команды было удобно, и все могли независимо двигаться по своим задачам.
Инициировал репозиторий, спроектировал базовую структуру папок проекта, настроил роутинг с защищенными маршрутами для неавторизованных пользователей.

### Технологии и инструменты

В этом проекте я впервые работал с React. Параллельно с разработкой изучал: `react-router-dom`, Zustand, Material UI, `useMediaQuery`, `useSearchParams`, `requestAnimationFrame`, основы CI/CD.
Стек проекта: **React + Material UI + Zustand + Axios + Vitest**, деплой на **Vercel** через **GitHub Actions**.

### Что разработал?

Я отвечал за заметную часть клиентской логики и интеграций:

- Создал общий `Layout` приложения, ориентируясь на макет, разработал `Header`, `Navigation`, `Footer` и дорабатывал их в ходе проекта
- Спроектировал и внедрил базовый роутинг приложения, учитывая защищенные и публичные маршруты, и предложил сценарий запуска квиза через маршрут `/practice/:id`
- Разработал и постоянно дорабатывал экран `Library` со сложным состоянием: фильтрация, карточки, статусы, интеграция с API, функциональность добавления в "Избранное"
- Разработал интерактивный тренажер `TrueFalseWidget` с таймером, флип-анимацией карточек, разными состояниями
- Создал `axiosInstance` с централизованной конфигурацией заголовков и интерцептором для автоматического обновления access-токена через refresh. До рефакторинга при ответе сервера `401` - пользователя сразу выбрасывало из приложения, каждые 10 минут. Разобрался и исправил проблему.
- Участвовал в улучшении локализации (`i18n`, `useLocale`) и устранении багов при смене языка, был лишний ререндер и перезагрузка сценариев: нашел проблемную зависимость, убрал лишние связи.
- Настроил и отладил CI/CD-поток, создав пайплайн на GitHub Actions — линтинг → тесты → билд → деплой на Vercel. Разобрался с Actions Secrets для хранения защищённых переменных, настроил `vercel.json` для корректной работы React Router (без него Vercel отдавал свою 404 страницу на прямых переходах).

### Команда и Code Review

Активно участвовал в ревью кода тиммейтов — тестировал, находил edge-cases, проблемы верстки, рассинхронизацию состояний в стейт-зависимых квизах, места где можно улучшить дизайн. Часто предлагал конкретные решения с объяснением причин.

Постоянно участвовал в жизни команды, обсуждая идеи, решения и видение проекта каждого участника. Выносил на обсуждение проблемы, возникающие в процессе разработки. Немалую часть решений валидировал через PR-цикл и командные обсуждения, слушая замечания.

## 3. Два личных Feature Component

### Feature Component #1 — Страница `/Library`

**Rich UI Screen**: Полноценная страница библиотеки квизов с комплексной логикой состояния. Вел ее развитие от моков до интеграции с реальным API, активно коммуницировал с backend - договаривались о контракте, объяснял как компонент устроен и какие данные нужны. Есть последовательная история развития по нескольким PR.

**Что реализовано:**

- Адаптивная сетка карточек квизов (от 320px до десктопа)
- Фильтрация по категории, типу тренажера, сложности, статусу прохождения и «Избранному»
- Пагинация через `useSearchParams` — параметры отражаются в URL
- Скелетон-загрузка и экран ошибки
- Обработка edge-cases: переполнение текстом, большое количество тегов
- Функция «Избранное» — лайк, позволяющий отложить тест и фильтровать по нему
- Интеграция с API бэкенда

Я старался продумать хороший UX и UI. Каждая карточка — не просто информация о квизе и кнопка запуска, но и показатель текущего состояния и статуса прохождения теста, учитывающая процент прохождения, и дающая дополнительные подсказки пользователю при наведении.

**Что было сложным и какие проблемы решал:**

- Интеграция фильтрации и пагинации с URL-параметрами — сначала не понимал, как синхронизировать состояние фильтров с URL-строкой так, чтобы запрос на сервер всегда соответствовал тому, что в адресной строке. Не один раз переделывал состояния фильтров.
- Функциональность «Избранного» — разбирался, показывать ли UI-изменение оптимистично или ждать ответа сервера, и как корректно обрабатывать ошибку. Изучил как это решается в больших проектах.
- Дизайн и верстка — продумывал какую информацию показывать, в каком виде, с учётом экранов вплоть до 320px.

### Feature Component #2 — Виджет True/False

**Complex Component:** Интерактивный тренажер со своим жизненным циклом, состояниями, таймером и анимациями. Продумывал, как сделать из простого виджета «Правда или Ложь» интересный тренажер — добавил визуально истекающее время, которое заставляет пользователя принимать решение быстро. Это отражает реальную подготовку к собеседованию, где часто нет времени на долгие размышления. После ответа или истечения времени таймер останавливается, карточка переворачивается и появляется объяснение — пользователь может спокойно проанализировать утверждение.

**Что реализовано:**

- Механика True/False с подтверждением ответа
- Кольцевой таймер на `requestAnimationFrame` с лимитом времени, зависящим от сложности теста
- CSS-анимации для визуальной обратной связи
- Кастомная стилизация без готовых MUI-компонентов там, где это требовалось
- Жизненный цикл вопроса, отдельный UI для таймаута

**Что было сложным и какие проблемы решал:**

- Продумывание механики — пользователь сразу видит верный ответ, нужно было корректно выстроить логику состояний для разных сценариев: обычный ответ, таймаут, навигация назад.
- Синхронизация анимации таймера с реальным временем — `setInterval` не синхронизируется с циклом рендеринга браузера. Пришлось искать решение через `requestAnimationFrame`.

### Дополнительно заявляю компонент Header (на случай дополнительной валидации)

Почему считаю его полноценной личной фичей:
Header - это не просто разметка с логотипом и ссылками. Компонент объединяет навигацию, пользовательское состояние, реализует два принципиально разных layout'а в зависимости от размера устройства, управляет состоянием Drawer, обрабатывает граничный пиксель в `useMediaQuery` и является точкой сборки кросс-функциональных систем: auth, i18n, тема.

По аналогии с примерами из условия — Widget Engine — Header выступает именно "движком" верхнего уровня, который управляет навигацией и переключает режимы отображения в зависимости от контекста.
